## 무엇을 만들었나

여러 패션 판매채널(편집숍, 브랜드 공식몰)에서 제품 가격을 자동 수집하고,
동일 제품의 채널별 최저가를 비교하는 웹 플랫폼.

- **기간**: 2026-02-25 ~ 2026-03-09 (약 2주)
- **Phase**: 1~8 (T-001 ~ T-140, 일부 미완)
- **최종 데이터**: 제품 80,000+개, 채널 137개, 브랜드 2,500+개
- **배포**: Railway (API + Worker) + Vercel (프론트엔드)
- **라이브 URL**: https://fashion-data-engine.vercel.app

---

## 왜 아카이빙했나

### 1. Railway 잦은 크래시

Railway 무료/스타터 플랜에서 API 서비스 + Worker 서비스를 동시에 운영하면서
메모리 초과 추정 크래시가 반복됐다. 알림이 계속 들어오는 상황에서
디버깅보다 새로 시작하는 것이 합리적이라 판단.

### 2. 구조가 너무 복잡해짐

2주 만에 Phase 1~8을 진행하면서 기능이 과도하게 쌓였다:

| Phase | 추가된 것 |
|-------|----------|
| 1~2 | 크롤링 기반, 채널/브랜드/가격 수집 |
| 3~4 | AI 쿼리 레이어, pgvector 시도 (실패) |
| 5 | 프론트엔드 폴리시 (17개 페이지) |
| 6 | 모니터링, 데이터 신선도 |
| 7 | 비즈니스 인텔리전스, 협업/디렉터 |
| 8 | OpenAPI, 웹훅, 구글 쇼핑 피드 |
| Phase 26~27 | Fashion Intel Hub (별도 4개 테이블, 지도, 타임라인) |

각 Phase마다 새로운 테이블, 스크립트, API가 추가되면서
유지보수 부담이 감당하기 어려운 수준이 됨.

### 3. 두 번의 방향 전환 시도, 모두 실패

**1차 피벗 시도 — LLM 병렬 검색 엔진** (`7564bfd`)

크롤링 중심 구조의 한계를 느끼고, OpenAI/Gemini/Claude API를 병렬 호출해
실시간으로 제품 정보를 검색하는 방식으로 전환 시도.
→ API 비용 문제로 포기.

**2차 피벗 시도 — 브라우저 자동화 퍼스널 쇼퍼** (`63bffd0`)

API 비용 없이 Playwright로 claude.ai/ChatGPT/Gemini 웹을
자동 제어해 검색 결과를 가져오는 방식으로 전환 시도.
→ uvicorn 실행 환경 문제(VIRTUAL_ENV 충돌)로 로컬에서도 실행 불가.
→ 피벗 과정에서 기존 크롤링 라우트가 전부 삭제되어 Railway 배포가 깨짐.

결국 `792a199`로 롤백 후 아카이빙 결정.

### 4. 핵심 가치 대비 복잡도 불균형

이 서비스의 핵심 가치는 **"같은 제품을 어디서 사는 게 제일 싸냐"** 하나였다.
그런데 Intel Hub, 지도, 타임라인, 비즈니스 인텔리전스 등
핵심과 거리가 먼 기능들이 너무 많이 붙었다.

## 잘 된 것들

- **product_key / normalized_key 교차채널 매칭**: 설계가 탄탄했고 실제로 잘 동작함
- **Shopify 크롤러**: `/products.json` 기반으로 80,000+개 제품 수집 성공
- **T-101 JPY 버그 수정**: 가격 이중부패 버그 완전 해결, 방어 로직 추가
- **Railway PostgreSQL 운영**: 2주간 데이터 쌓아둔 것은 자산
- **Claude Code + Codex 협업 구조**: AGENTS.md 기반 핸드오프는 효율적이었음
